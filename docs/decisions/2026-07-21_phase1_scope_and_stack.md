## 背景

基线规格（v2.0）定义了完整的交易宪法、Scanner 规则、模拟交易阶段与工程模块，但未锁定实现语言、存储、数据源与交付切片方式。  
在进入编码前，需要把会显著影响工期与审计边界的选择写成正式决策，避免仅存在于对话中。

---

## 决策

### 1. 技术栈

**采用：**

- Python 3.11+（运行环境允许更高版本）
- 本地存储：价格/特征以 Parquet 为主；信号、订单、持仓、版本以 SQLite 为主
- 数据源：可插拔 `DataProvider`；Phase 1 先使用免费/低成本行情，预留付费源适配
- 配置：YAML 冻结 V1 参数 + 校验（实现阶段再定具体库）
- 任务入口：CLI（如 `run_daily --as-of`）

**不采用（Phase 1）：**

- 以生产级分布式数据库或消息队列为一等依赖
- 自动真实下单通道

### 2. 实施优先级

> **2026-07-22 起：** 本节由 [2026-07-22_phase1_mvp_slicing_v1_1.md](./2026-07-22_phase1_mvp_slicing_v1_1.md) **替代**。  
> 现行策略为 **两阶段 MVP（Signal → Risk+Paper）**，不再以「一次打通至含 Risk 的 Shadow」为唯一切片。

**历史原文（已替代，保留审计）：**

**采用：端到端 MVP 优先。**

顺序原则：

1. 先打通：数据 → Scanner → 风险计划 → 日报 → Shadow（不建仓）
2. 再补：Paper 成交与出场、熔断
3. 其后：完整回测 harness、敏感性、消融、Walk-Forward

**不采用：** 在出现首个可复现日更信号之前，先完整建设全部研究验证设施。

### 3. 基本面数据范围（V1）

> **2026-07-22 起：** 落地方式由 [2026-07-22 ADR](./2026-07-22_phase1_mvp_slicing_v1_1.md) **修订** — V1 基本面**优先作风险过滤**，不以 TotalScore 主权重驱动。  
> 下列「精简 + 可缺席」原则仍有效；执行细节以新 ADR + 冻结 YAML 为准。

**采用：精简基本面。**

- 以价格动量、趋势、相对强度、Setup/突破结构为主
- `FundamentalScore` 允许缺失，必须标记为 `missing` / N/A
- **禁止**把缺失数据静默填成 0 或有利默认值
- 完整 point-in-time 财报字段与修订历史不作为 Phase 1 MVP 阻塞项
- 若后续接入财报日历，优先服务「事件风险过滤」，再服务评分

---

## 理由

1. **个人 Phase 1 规模**：SQLite + Parquet 足以支撑日线股票池与审计表，降低运维噪音，把注意力放在规则正确性与可复现上。
2. **可插拔数据源**：免费源便于尽快验证流水线；规格要求的数据正确性可通过 validation fail-closed 与后续更换 provider 满足，而不把第一刀焊死在某一供应商。
3. **MVP 优先**：规格强调「先定义系统，再验证系统」。没有可重复的信号与审计快照，回测基础设施容易与真实日更逻辑分叉。
4. **基本面精简**：规格已写明 V1 基本面不宜过多硬门槛，且不同行业财务结构差异大；缺失即标记符合「数据正确性高于模型复杂度」。

---

## 后果

### 正面

- 更快进入 Stage A Shadow 的工程就绪状态
- 审计链可从第一天建立（版本、配置、信号原因码）
- 基本面数据采购与 point-in-time 工程可独立演进

### 代价 / 风险

- 免费行情在复权、停牌、退市、成分股历史上可能不足 → 正式策略结论前必须披露限制
- 无历史成分股时，任何「回测收益」不得当作晋级证据
- `FundamentalScore` 权重仍在 TotalScore 中时，缺失处理规则必须写进 config 并保持稳定

### 后续必须遵守

- 更换数据源、升级基本面、改变 MVP 切片，均需新的 decision 文档，并评估是否 bump 策略版本
- 执行中的模拟实验不得与研究改参结果混统计

## 备选方案（未采纳）

| 方案 | 未采纳原因 |
|------|------------|
| Python + PostgreSQL + 指定付费数据商 | 更接近生产，但提高 Phase 1 启动成本；可在数据成为瓶颈后再升级 |
| 研究严谨优先（先全套回测/WF/消融） | 延迟可审计日更闭环；双轨逻辑风险更高 |
| V1 即完整基本面 + 财报日历 | 数据与 point-in-time 工程量大，阻塞 Scanner 主路径验证 |
| 仅财报日历、完全不做基本面分 | 可作为更瘦 MVP，但当前选择保留「可缺席评分」扩展位，与规格 TotalScore 结构兼容 |
